01 · Finding

Today's Focus surfaces what's there. Nothing checks what's missing.

  • Ryan's work lives in subtask names, not assignee fields. The briefing reads parent + subtask structure, not just top-level tasks.
  • Asana custom-field metadata is essentially empty across all 10 client projects. Industry, AM, state, complexity, service mix — none of it lives in Asana.
0of10
Asana client projects with structured client metadata. All of it lives in the Notion Client Register and Services databases. Asana is the execution surface; Notion is the source of truth.
  • This means: whether the right tasks even exist in Asana cannot be verified from inside Asana. It can only be verified by reading the Notion Service Templates milestone chains and comparing against what's present per client project.
  • No system does this today. Tasks go missing silently. Today's Focus has no way to know — it surfaces a 3–5 priority list from a task graph that may already be incomplete.

And a third finding · chain progress is invisible

  • Even when a service's tasks exist in Asana, Asana has no native concept of "this BAS is 2 of 10 milestones done with 3 days to go." Each milestone is just a task in a list. There's no shape to the chain.
  • That shape is what tells Ryan which service is actually at risk. 2-of-10 due Friday needs his attention now. 9-of-10 due Friday is a 30-minute review on Friday morning.
  • The Notion Service Templates already define the canonical milestone chain for each of the 19 services. This system makes that chain a first-class structure in Asana — and uses chain progress as a top-tier scoring signal in Today's Focus.
  • Implication. Three jobs: (1) materialise every service's milestone chain in Asana with computed dates and role-derived assignees, (2) make chain progress visible and quantifiable, (3) surface the result in Today's Focus with reasoning Ryan can trust.
02 · Two layers

Verify the data · surface what matters today.

Two daily jobs. The reconciler makes sure every client × service has the right tasks in Asana — by reading the milestone chain from Notion's Service Templates and creating any that are missing. The briefing takes that verified task list, scores every open subtask along five inputs (calendar, service threshold, chain progress, Claude-reasoned effort, sensitivity) and surfaces what Ryan should look at right now.

  • Reconciler runs at 06:00. Briefing refreshes at 08:00 and 16:00.
  • The reconciler runs first so the briefing operates on a complete graph. If the reconciler fails, the briefing still runs — it just operates on yesterday's state.

Layer 1 · Reconciler — 06:00 AEST

  • For each client × subscribed service, expands the Notion Service Template milestone chain into a target structure: parent task = service instance · subtasks = milestones in order.
  • Computes each subtask's due date from anchor_date + days_offset. Anchor is set per service-instance (e.g. Quarterly BAS Q1 anchors at the lodgement deadline; weekly bookkeeping anchors at the Monday).
  • Resolves each subtask's assignee from role → person: offshore_lead → Sailesh / Sriram (from Client Register), AM → Ryan, accountant → derived per service.
  • Compares the target structure against what's actually in Asana (parent + subtask graph). Missing parents and missing subtasks both surface.
  • Each gap becomes a row in Pending Reconciliation grouped by service-instance. One Approve tick = create the missing parts of that whole chain. Ryan does not approve 10 individual milestones — he approves "EMPAC Q1 BAS, missing 3 subtasks, create them."
  • Detect + Suggest, not Auto-Create. The reconciler never writes to Asana without explicit human approval.
  • Hybrid matching. New subtasks are stamped with a hidden Milestone Template ID Asana custom field. Future runs match by ID — deterministic, immune to renames. Legacy subtasks fall back to fuzzy name match (similarity ≥ 0.85). Matches below 0.92 are flagged as low-confidence.
  • Partial-chain handling. If 7 of 10 milestones already exist in Asana from legacy work, the reconciler creates only the 3 missing — never destroys legacy state. Pre-write similarity check guards against duplicates.

Layer 2 · Today's Focus — 08:00 / 16:00 AEST

  • One briefing. Single marker block on Ryan's existing AM Workspace page. Same shape both times — refreshed at 08:00 with the start-of-day picture, refreshed at 16:00 with where things stand at end of day. Not a separate "morning view" and "afternoon view" — one surface, two refreshes.
  • The block always shows: 🔴 critical · 🟠 this week · 🟡 on the radar · ✅ cleared since last refresh — sorted by service-specific urgency state.
  • One atomic marker-block write per refresh on Ryan's existing AM Workspace page.

Why two layers and not one?

The reconciler and the briefing answer two different questions. Conflating them would mean every briefing run has to also do reconciliation — doubling its runtime, coupling its failure modes, and confusing what the system is actually doing.

  • Reconciler asks: "Are the right tasks even in Asana?" Reads Notion templates · creates missing chain pieces. Once a day, before anyone is at their desk.
  • Briefing asks: "Of all the tasks Ryan has, which matter most right now?" Reads the verified Asana state · ranks · surfaces. Twice a day, when Ryan checks in.
  • The reconciler runs first so the briefing operates on a complete graph. If the reconciler fails, the briefing still runs — it just operates on yesterday's state.

The urgency model · how a task gets a state

Every open subtask gets exactly one state — 🔴 / 🟠 / 🟡 / 🟢 — produced by combining five inputs. Four are deterministic. One is reasoned by Claude. Together they answer "is this urgent and can Ryan actually fit it in today?"

Input What it tells the system Source
1. Days until due How much calendar runway is left. Negative = overdue. Asana
2. Service threshold (yellow_at / orange_at / red_at) What "urgent" means for this kind of work. A monthly report turns orange at T-3; a BAS turns orange at T-7. Each service has its own rhythm, defined once in the Notion Service Template. Notion
3. Chain progress (subtasks_done / total · chain_pressure) Whether the milestone chain will actually finish in time at the current pace. 1 of 10 done at T-7 = chain won't land. 9 of 10 done at T-3 = it's fine. Asana parent + subtask graph
4. Effort estimate (difficulty × hours) Reasoned by Claude from the subtask name, service template description, and any history on the task. Returns a difficulty band (S / M / L / XL) and an estimated hours figure. Cached on the task — only re-reasoned when the task changes. A 30-min task at T-3 reads differently from a 6-hour task at T-3. Claude · cached on Asana task
5. Sensitivity flag If the client is flagged sensitive in the Client Register, the state is bumped up one level (yellow → orange, orange → red). One-way; never bumps down. Notion · Client Register

The four states

State What it means Example
🔴 Critical Act today or it slips. Past red_at, overdue, or chain progress + effort estimate say it can't finish at current pace. Monthly bookkeeping at T-1 · BAS at T-2 · any task overdue · 6h of work remaining at T-1
🟠 This week Past orange_at. Should land in the next few days; needs planning. Monthly report at T-3 · BAS at T-7 · weekly payroll at T-1
🟡 On the radar Past yellow_at. Early visibility — Ryan should know it exists, not act yet. Monthly report at T-7 · BAS at T-14 · annual statements at T-30 · low-priority task at T-5
🟢 Background Inside the threshold window. Collapsed by default. Anything before its service's yellow_at

Worked example · two BAS subtasks at T-7

Two subtasks, same service, same days-until-due. They get different states because chain progress and effort estimate differ:

  • EMPAC · BAS · "Reconcile bank feed" · T-7 · 1 of 10 chain subtasks done · Claude estimates difficulty M / 3h. → 🟠 orange. The BAS template says T-7 is yellow, but chain progress is too far behind, so the state escalates one band.
  • Hudson · BAS · "Final review" · T-7 · 8 of 10 done · Claude estimates difficulty S / 30min. → 🟢 green. T-7 would normally be yellow, but progress and effort say it's fine; collapsed.

state = compute_state(
  days_until_due,
  service_template.thresholds,  # {yellow_at: 7, orange_at: 3, red_at: 1}
  chain_progress,   # {done: 1, total: 10, pressure: subtasks_remaining / days_until_due}
  effort_estimate,   # {difficulty: "M", hours: 3} — reasoned by Claude, cached on the task
  sensitivity_flag   # bumps state up one level if true
)

# returns: red | orange | yellow | green

How effort estimation works

  • One Claude call per new or changed subtask. Inputs: subtask name, parent service, service template description, any task description / comments, prior similar-subtask estimates. Outputs: difficulty band (S / M / L / XL) and estimated hours.
  • Cached on the Asana task in two new custom fields: Difficulty (dropdown: S/M/L/XL) and Est. Hours (number). Visible to Ryan; editable — manual edits are respected and not overwritten.
  • Re-reasoned only when the task changes (rename, description edit, comment added). Steady-state cost is near zero — most tasks keep the same estimate week-over-week.
  • Why it matters for state. Effort × chain progress tells the system whether the chain can finish in time, not just whether the deadline is close. A monthly report at T-7 with 12h of remaining work is critical even though the calendar says yellow.

System MCP connector Auth What this system does with it

B · Notion write pattern

  • Atomic marker-block rewrites — the "🎯 Today's Focus" block is replaced as a unit, never partially rendered.
  • One page only: Ryan's existing AM Workspace page. The marker block is rewritten at 08:00 and 16:00.
  • Reconciler writes to Pending Reconciliation DB (rows) — not to any page block.
  • Today's Focus block: heading "🎯 Today's Focus" on Ryan's workspace page. Found by ID. PATCH /v1/blocks/{marker_id}/children atomically replaces children.
  • Pending Reconciliation: standard DB row writes via POST /v1/pages with database parent.
  • Crash-safe: commits fully or not at all. No half-rendered state.
  • Failure mode: previous render persists until next successful run. Block header shows "⚠️ Last run failed at HH:MM."

Mid-run crash Idempotent — next run computes desired state from current and writes.
