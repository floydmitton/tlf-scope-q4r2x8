01 · Finding

Today's Focus surfaces what's there. Nothing checks what's missing.

  • v18 finds Ryan's work in subtask names rather than assignee fields. Carries forward unchanged.
  • v19 adds a second finding: Asana custom-field metadata is essentially empty across all 10 client projects. Industry, AM, state, complexity, service mix — none of it lives in Asana.
0of10
Asana client projects with structured client metadata. All of it lives in the Notion Client Register and Services databases. Asana is the execution surface; Notion is the source of truth.
  • This means: whether the right tasks even exist in Asana cannot be verified from inside Asana. It can only be verified by reading the Notion Service Templates milestone chains and comparing against what's present per client project.
  • No system does this today. Tasks go missing silently. Today's Focus has no way to know — it surfaces a 3–5 priority list from a task graph that may already be incomplete.

And a third finding · chain progress is invisible

  • Even when a service's tasks exist in Asana, Asana has no native concept of "this BAS is 2 of 10 milestones done with 3 days to go." Each milestone is just a task in a list. There's no shape to the chain.
  • That shape is what tells Ryan which service is actually at risk. 2-of-10 due Friday needs his attention now. 9-of-10 due Friday is a 30-minute review on Friday morning.
  • The Notion Service Templates already define the canonical milestone chain for each of the 19 services. v19 makes that chain a first-class structure in Asana — and uses chain progress as a top-tier scoring signal in Today's Focus.
  • Implication. v19 has three jobs: (1) materialise every service's milestone chain in Asana with computed dates and role-derived assignees, (2) make chain progress visible and quantifiable, (3) surface the result in Today's Focus with reasoning Ryan can trust.
02 · Two layers

Verify the data · surface what matters today.

  • Two layers. The reconciler runs first; Today's Focus runs on top of the verified state.
  • If the reconciler fails, Today's Focus still runs — it just operates on yesterday's state.

Layer 1 · Reconciler — 06:00 AEST

  • For each client × subscribed service, expands the Notion Service Template milestone chain into a target structure: parent task = service instance · subtasks = milestones in order.
  • Computes each subtask's due date from anchor_date + days_offset. Anchor is set per service-instance (e.g. Quarterly BAS Q1 anchors at the lodgement deadline; weekly bookkeeping anchors at the Monday).
  • Resolves each subtask's assignee from role → person: offshore_lead → Sailesh / Sriram (from Client Register), AM → Ryan, accountant → derived per service.
  • Compares the target structure against what's actually in Asana (parent + subtask graph). Missing parents and missing subtasks both surface.
  • Each gap becomes a row in Pending Reconciliation grouped by service-instance. One Approve tick = create the missing parts of that whole chain. Ryan does not approve 10 individual milestones — he approves "EMPAC Q1 BAS, missing 3 subtasks, create them."
  • Detect + Suggest, not Auto-Create. The reconciler never writes to Asana without explicit human approval.
  • Hybrid matching. New subtasks are stamped with a hidden Milestone Template ID Asana custom field. Future runs match by ID — deterministic, immune to renames. Legacy subtasks fall back to fuzzy name match (similarity ≥ 0.85). Matches below 0.92 are flagged as low-confidence.
  • Partial-chain handling. If 7 of 10 milestones already exist in Asana from legacy work, the reconciler creates only the 3 missing — never destroys legacy state. Pre-write similarity check guards against duplicates.

The chain pressure signal

For every open service-instance Ryan has visibility on, the system computes:

chain_pressure = subtasks_remaining / max(days_until_chain_due, 0.5)

  • 2 of 10 done · due Friday (3 days) → 8 / 3 = 2.67 (HIGH) — Ryan needs to act now or the chain won't finish.
  • 9 of 10 done · due Friday (3 days) → 1 / 3 = 0.33 (LOW) — coast, finish Friday morning.
  • Chain pressure is the dominant input to the priority score:

priority_score =
  urgency × 3   # 1 / max(days_until_due, 0.5)
+ chain_pressure × 4   # subtasks_remaining / days_until_due
+ criticality × 2   # service tier (BAS/payroll = 1.0, bookkeeping = 0.5)
+ sensitivity × 1   # client flag from Client Register
+ slipped × 1.5   # on this morning's picks, not done

  • The Today's Focus block surfaces both the score and the reasoning — "EMPAC Q1 BAS · 2 of 10 done · 3 days to go · chain pressure 2.67" — so Ryan can challenge or trust the ranking.
  • Weights are tunable in a Notion config DB Ryan owns. No code changes needed to retune.

D · Error handling

Failure Outcome
Asana API down Run aborts cleanly. Today's Focus block shows ⚠️ Asana unavailable, retry at HH:30.

  • Iteration 6 · Meetings → Commitments — capture meeting outputs (Zoom + manual). Populates the Commitments DB. Folds into Today's Focus as another input source — no architectural change to v19.
  • Iteration 7 · Operational risks layer — payroll dependencies, BAS lodgement deadlines, ATO/ASIC dates surfaced as ambient warnings on the briefing block.
  • Iteration 8 · Client workspaces — clients see their own service progress in a read-only Notion view. Strict isolation. Requires the security model.
  • Iteration 9 · External system sync — Xero (bookkeeping/payroll) · XPM (contact sync) · NowInfinity / ASIC (entity register). Each additive — no destructive overwrite.
  • Iteration 10 · Warehouse + inference layer — BigQuery shadow logger captures every run event from day 1 of v19 if Sam wants the warehouse foundation now (additional ~10h, decision deferred).

What this is not

  • Not Manoli yet. Ryan only this iteration. Manoli is Iteration 5 — a clone with a different filter, not new engineering.
  • Not commitments yet. Meetings → Commitments capture is its own iteration. Once it ships, commitments slot into Today's Focus as an input alongside Asana and email.
  • Not a two-way sync. Asana ↔ Notion bidirectional sync is deferred. The reconciler is one-way (Notion → Asana) and gated on human approval.
  • Not a data warehouse. No BigQuery yet. State lives in Notion.
  • Not an AI inference layer. Claude is used tactically for ranking and risk lines, not as a general query interface.
