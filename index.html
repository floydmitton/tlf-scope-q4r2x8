01 · Finding

Today's Focus surfaces what's there. Nothing checks what's missing.

  • v18 finds Ryan's work in subtask names rather than assignee fields. Carries forward unchanged.
  • v19 adds a second finding: Asana custom-field metadata is essentially empty across all 10 client projects. Industry, AM, state, complexity, service mix — none of it lives in Asana.
0of10
Asana client projects with structured client metadata. All of it lives in the Notion Client Register and Services databases. Asana is the execution surface; Notion is the source of truth.
  • This means: whether the right tasks even exist in Asana cannot be verified from inside Asana. It can only be verified by reading the Notion Service Templates milestone chains and comparing against what's present per client project.
  • No system does this today. Tasks go missing silently. Today's Focus has no way to know.
  • Implication. Today's Focus needs a layer beneath it that guarantees the comparison happens daily. That layer is the Reconciler.
02 · Three layers

Verify the data · organise the day · surface the priorities.

  • A stack of trust. Each layer is independent and runs on its own schedule.
  • If one fails, the others are unaffected.

Layer 1 · Reconciler — 06:00 AEST

  • For each client × subscribed service, compares what should be in Asana (the Notion Service Templates milestone chain) against what is.
  • Anything missing becomes a row in a Pending Reconciliation Notion DB with an Approve checkbox.
  • When Ryan ticks Approve, the next 06:00 run creates the Asana task with the right assignee, due date, and parent section.
  • Detect + Suggest, not Auto-Create. The reconciler never writes to Asana without explicit human approval.
  • Hybrid matching. New tasks are stamped with a hidden Milestone Template ID Asana custom field. Future runs match by ID — deterministic, immune to renames. Legacy tasks fall back to fuzzy name match (similarity ≥ 0.85). Matches below 0.92 are flagged as low-confidence.

Layer 2 · AM/PM Wrap-Up — 09:00 / 17:00 AEST, per AM

  • AM (09:00). Pull every incomplete task assigned to the AM across all 10 projects, plus open Commitments. Score each by urgency, size, service criticality, sensitivity, slippage. Render top 10 to the AM's Daily Briefing page with risk lines and italic relationship context.
  • PM (17:00). Replace the AM block with two buckets — 🔴 Slipped from this morning (items on AM Wrap-Up not done) and ⚠️ New today, still open (items that arrived after 09:00).
  • Same atomic marker-block write pattern as v18.

Layer 3 · Today's Focus — 08:00 / 16:00 AEST

  • Unchanged from Scope v18. Pulls ~45 candidates · 4 deterministic rules · LLM ranks · 3–5 lines on Ryan's AM Workspace page.
  • The Reconciler runs before it, so Today's Focus operates on a verified task graph.

06 · How it runs

Integration model and a 06:00 walkthrough.

  • Same runtime as v18 — Cowork. No new infrastructure.
  • Two new playbooks beside Today's Focus: the Reconciler and the Wrap-Up engine.
  • Walkthrough below is the 06:00 reconciler run on Tuesday 28 April · synthetic data · total elapsed ~40 seconds.

A · Integration model

System MCP connector Auth What v19 adds
Asana asana-mcp OAuth on Floyd's account (same as v18) Reads Milestone Template ID custom field · writes new tasks for approved rows · stamps template IDs.
Notion notion-mcp Same integration token as v18 Reads Service Templates milestone chains · writes Pending Reconciliation rows · maintains Reconciliation Log + AM Briefing State · atomic marker-block rewrites on Daily Briefing pages.

B · Notion write pattern

  • Same atomic marker-block pattern as v18.
  • Each AM gets a new Daily Briefing page in the Nine Advisory HQ teamspace. One page per AM, one marker block per page.
  • Reconciler writes to Pending Reconciliation DB (rows) — not to any page block.
  • Daily Briefing block: heading "🌅 AM Wrap-Up" / "🌙 PM Wrap-Up" on each AM's Daily Briefing page. Found by ID. PATCH /v1/blocks/{marker_id}/children atomically replaces children.
  • Pending Reconciliation: standard DB row writes via POST /v1/pages with database parent.
  • Crash-safe: commits fully or not at all. No half-rendered state.
  • Failure mode: previous render persists until next successful run. Briefing page header shows "⚠️ Last run failed at HH:MM."

D · Error handling

Failure Outcome
Asana API down Run aborts cleanly. Briefing block shows ⚠️ Asana unavailable, retry at HH:30.
Notion API down Atomic writes fail-fast. No partial state. Next run resumes.
Claude down Wrap-up renders without LLM context lines. Header notes "↳ context unavailable." Reconciler unaffected (deterministic).
Mid-run crash Idempotent — next run computes desired state from current and writes.
Matcher false positive (low-confidence) Surfaced on briefing page header as warning + audited weekly. Tightens similarity threshold from flags.
Matcher false negative (duplicate detect) Pre-write similarity check ≥ 0.85 against existing tasks → skip + log.

E · Auditability

  • Reconciliation Log — per reconciler run · clients scanned, milestones expected/present/missing, approvals applied, low-confidence count, runtime, status.
  • Run Log (extended from v18) — per wrap-up + Today's Focus run · source counts, surfaced count, runtime, errors.
  • Decision Log (from v18, extended) — covers wrap-up scoring decisions: which items scored what, why each was surfaced or shelved.
  • All Ryan-visible. Filterable. 90-day rolling retention.

  • Iteration 9+ · Warehouse + inference layer — BigQuery shadow logger captures every run event from day 1 of v19 if Sam wants the warehouse foundation now (additional ~10h, decision deferred).

What this is not

  • Not a two-way sync. Asana ↔ Notion bidirectional sync is deferred. The reconciler is one-way (Notion → Asana) and gated on human approval.
  • Not a data warehouse. No BigQuery yet. State lives in Notion.
  • Not an AI inference layer. Claude is used tactically for ranking and risk lines, not as a general query interface.
  • Not a CRM replacement. Interaction Log and Commitments are read-only here.
