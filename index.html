Today's Focus alone surfaces what matters today, but it silently assumes the right tasks exist in Asana to begin with. They don't. When a client signs up for a service in Notion, no system today guarantees the corresponding milestone tasks land in Asana. Today's Focus can only surface what's there — anything missing stays missing.

Scope v19 fixes the layer underneath, then adds a daily lens for the AM, then surfaces priorities. Three layers of focus:

  1. Reconciler — every morning at 06:00, check that every active client's services have their milestone tasks in Asana. If something's missing, draft it and ask the AM to approve.
  2. AM/PM Wrap-Up — every morning at 09:00, hand the AM a single page with their full plate, prioritised, with what's at risk if they don't act. At 17:00, replace it with what slipped and what landed today.
  3. Today's Focus — Scope v18 retained verbatim. Twice-daily 3–5 things that matter most, mixed across email, Asana, and commitments.

A stack of trust: verify the data → organise the day → surface the priorities. Sam sees a complete daily operating system for Ryan and Manoli, not a feature.

Engineering effort: ~65 hours over 4 weeks (3 weeks build + 1 week shadow/tune). Ryan's effort: ~3 hours. Stack: Cowork + Asana MCP + Notion MCP. No new infrastructure.

01 Finding

Two findings drive this scope.

Finding 1 (carried from v18). Ryan's work assignments live in subtask names rather than Asana's assignee fields. Standard "assignee = Ryan" filters return 2 tasks despite Ryan touching dozens weekly. Offshore teams embed ownership as text patterns — ↳ Ryan to review — instead of structured assignment.

Finding 2 (new, motivates v19's foundation layer). Asana custom-field metadata across the 10 client projects is essentially empty. Industry, account manager, state, complexity, service mix — none of it lives in Asana. All of it lives in the Notion Client Register and Services databases. Asana is the execution surface; Notion is the source of truth.

This means: whether the right tasks exist in Asana cannot be verified from inside Asana. It can only be verified by reading the Notion Service Templates milestone chains and comparing them against what is present in each client project. No system does this today. Tasks go missing silently, and Today's Focus has no way to know.

Implication. Today's Focus needs a layer beneath it that guarantees the comparison happens daily. That layer is the Reconciler.

02 Three layers

Layer 1 — Reconciler · 06:00 AEST

For each client × subscribed service, compare what should be in Asana (the Notion Service Templates milestone chain) against what is in Asana. Anything missing becomes a row in a Pending Reconciliation Notion DB with an Approve checkbox. When Ryan ticks Approve, the next 06:00 run creates the Asana task with the right assignee, due date, and parent section.

Detect + Suggest, not Auto-Create. The reconciler never writes to Asana without explicit human approval. Two reasons: trust — false positives in the matcher would otherwise pollute the workspace; and control — Ryan reads the suggested task name, due date, and assignee before agreeing.

Hybrid matching. Tasks the reconciler creates are stamped with a hidden Milestone Template ID Asana custom field (one-time setup, ~5 min per project). Future runs match by ID — deterministic, immune to renames. Legacy tasks (created before the MVP) are matched by normalised name with similarity ≥ 0.85. Any match below 0.92 is flagged as low-confidence on the briefing page for human review.

Layer 2 — AM/PM Wrap-Up · 09:00 / 17:00 AEST, per AM

AM Wrap-Up (09:00). For each AM, pull every incomplete Asana task assigned to them across all 10 client projects, plus their open Commitments. Score each by urgency, size, service criticality, client sensitivity, and slippage. Render the top 10 to the AM's Daily Briefing page with: priority emoji, deadline tag, subtask count, "what's at risk if you don't touch this today" line (LLM-generated from service template + Interaction Log), and italic relationship context (last contact, sentiment).

PM Wrap-Up (17:00). Replace the AM block with a two-bucket view. 🔴 Slipped from this morning — items that were on the AM Wrap-Up but aren't completed. ⚠️ New today, still open — items that became due, assigned, or relevant after 09:00 and aren't completed. Plus a count: "✅ Cleared today: X of Y morning items."

The block on the Daily Briefing page is atomically rewritten three times daily (AM at 09:00, PM at 17:00, AM again next morning). Same marker-block pattern as v18.

Layer 3 — Today's Focus · 08:00 / 16:00 AEST

Unchanged from Scope v18. Pulls ~45 candidates from Asana + Email Triage + Commitments, filters via four deterministic rules, ranks via Claude Sonnet 4.6, surfaces 3–5 lines on Ryan's existing AM Workspace page. The Reconciler runs before it, so Today's Focus operates on a verified task graph.

04 Pipeline

Five daily runs, each independent. If one fails the others are unaffected.

06:00 RECONCILER → Pending Reconciliation rows + Asana creates (approved) 08:00 TODAY'S FOCUS (AM) → Ryan's AM Workspace block (v18, unchanged) 09:00 AM WRAP-UP × 2 AMs → Daily Briefing pages, AM block 16:00 TODAY'S FOCUS (PM) → Ryan's AM Workspace block (v18, unchanged) 17:00 PM WRAP-UP × 2 AMs → Daily Briefing pages, PM block (replaces AM)

Daily totals: ~459 API calls, 6 LLM calls, ~7 minutes of compute. All within free-tier limits.

Reconciler run anatomy

  1. Read Notion config: Client Register, Services, Service Templates, Pending Reconciliation (checked rows)
  2. Read Asana: tasks per active client project, with Milestone Template ID custom field
  3. Match in-process: ID-first, then fuzzy name fallback
  4. Apply approved: create new Asana tasks, stamp template IDs, mark Notion rows Created
  5. Write missing: new Pending Reconciliation rows for newly-detected gaps
  6. Log run

AM Wrap-Up run anatomy

  1. Read Notion: yesterday's AM Briefing State, Client Register, Commitments, Interaction Log, Service Templates
  2. Read Asana: incomplete tasks assigned to AM
  3. Score in-process: priority formula, top 10
  4. One Claude call: per-item risk lines + italic context
  5. Write Notion: AM Briefing State row + atomic block rewrite + Run Log

PM Wrap-Up run anatomy

  1. Read this morning's AM Briefing State row
  2. Re-read Asana: completion status of morning items + new items since 09:00
  3. Bucket: slipped vs new
  4. One Claude call: short reasoning per item
  5. Write Notion: atomic block replace + Run Log

06 How it runs

Runtime

Cowork (Anthropic's agentic runtime). No backend service, no Postgres, no Lambda, no separate cron. Same stack as Scope v18.

Connectors

  • asana-mcp — OAuth via Cowork. ~1-hour token, refreshed at start of each run.
  • notion-mcp — Notion integration token. Reads config + data DBs, writes via atomic block PATCH.
  • Claude Sonnet 4.6 — one call per wrap-up run; the reconciler is deterministic and uses no LLM.

State

All state lives in Notion DBs. No external database, no warehouse, no shadow logger (deferred to a future iteration). Notion is human-readable, queryable, and at Nine Advisory's scale — well under the size where this becomes a constraint.

Error handling

FailureOutcome
Asana API downRun aborts cleanly. Briefing block shows "⚠️ Asana unavailable, retry at HH:30."
Notion API downAtomic writes fail-fast. No partial state. Next run resumes.
Claude downWrap-up renders without LLM context lines. Header notes "↳ context unavailable."
Asana token refresh failsRun aborts. Slack DM to Floyd (operator).
Mid-run crashIdempotent — next run computes desired state from current and writes.
Matcher false positive (low-confidence)Surfaced on briefing page header as warning + audited weekly.
Matcher false negative (duplicate detect)Pre-write similarity check ≥ 0.85 against existing tasks → skip + log.
Cowork run doesn't fireFallback retry 30 min later. Heartbeat alert to Floyd.

The atomic marker-block pattern from v18 means partial briefing blocks are impossible — either the previous block stays, or the new one fully replaces it.

Auditability

Every run logs to a Notion DB:

  • Reconciliation Log — per reconciler run: clients scanned, milestones expected/present/missing, approvals applied, runtime, status.
  • Run Log — per wrap-up + Today's Focus run: source counts, surfaced count, runtime, errors.
  • Decision Log (from v18) — extended to also cover wrap-up scoring decisions.

Filterable views: "last 7 days, status ≠ ok," "this week, low-confidence matches." Ryan-visible. 90-day rolling retention with auto-archive.

— What this is not

To set expectations explicitly:

  • Not a two-way sync. Asana ↔ Notion bidirectional sync is deferred. The reconciler is one-way (Notion → Asana) and gated on human approval.
  • Not a data warehouse. No BigQuery, no Postgres, no separate analytics layer. State lives in Notion. The warehouse is a future iteration.
  • Not an AI inference layer. Claude is used tactically for ranking and risk-line generation, not as a general query interface over Nine Advisory data.
  • Not a CRM replacement. The Interaction Log is read-only here; commitments are read-only here. They feed the briefing but aren't restructured.

These are deliberate boundaries. Each is a real future iteration with its own scope. v19 is the foundation that makes all of them coherent.
